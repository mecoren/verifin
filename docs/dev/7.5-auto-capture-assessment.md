# 7.5 自动记账 — 可行性评估与完整方案（第三版 · 定稿方向）

> 状态：**7.5a（NLS 通知）Dart 层 + 原生已实现，待真机验收**；7.5b（无障碍补微信）与截屏方案不做。
> 关键调整（据真实使用反馈）：**解析改用 AI 必需，不写按银行/App 正则**；**后台直接落账**（不再确认弹窗，每条可 App 内复查/删）；主推云端模型（本地模型效果不足）；开关以「已配 AI」为硬门槛；通知先过「含数字」再调 AI 省 Token。
> 已实现（待真机）：`lib/app/auto_capture/*`（设置/前置过滤/AI 通知解析/管线/服务）+ 设置页 `AutoCaptureSettingsPage`（Alpha 提示）+ 原生 `PaymentNotificationListenerService`/`AutoCaptureBridge` + `verifin/app` 通道方法。**未做**：应用被杀时也自动落账（需无头引擎/WorkManager 触发 Dart，当前为「后台入队 + 回前台解析落账」）。
> 本文档为内部技术评估，正文保留中文（不进 ARB）。

## 背景

来源于一批真实用户反馈，核心痛点是「坚持不了每次点进去手工记账，很割裂」，呼声最高的诉求是**自动记账**。调研主流做法后定方向：**NLS（通知监听）+ 无障碍**双通道，纯 opt-in，砍掉截屏方案。

## 关键前提：本应用不上架 Google Play

Veri Fin 的 Android 安装包**只经 GitHub Releases 分发**（见 `CLAUDE.md`）。因此「Google Play 敏感权限政策审核严、会被下架」**基本不成立**。真正剩下的约束是：**用户信任**、**版面/文案脆弱性**、**微信支付宝 ToS**。

## 调研：别人是怎么做自动记账的

| 结论 | 说明 |
|------|------|
| **半自动是共识** | 主流 App 都是「自动抽金额/商户/时间 → 用户选分类 → 确认落账」，**没人自动落账**。与我们「绝不自动落账」红线一致。 |
| **NLS 与无障碍分工互补** | 支付宝/银行/云闪付通知**带金额**，NLS 正则可解析；**微信通知常不带金额**，需无障碍读「支付成功」页。所以两个都要。 |
| **最大工程坑是国产 ROM 保活** | 无障碍服务在小米/华为/OPPO/vivo 上会被后台杀。需**前台服务 + 常驻通知 + 电池优化白名单 + 自启动**引导才能长活。稳定性天然有限，需向用户诚实告知。 |
| **别人多用正则，我们改用 AI** | 别人靠正则抽金额（如 `([0-9]+|[0-9]{1,3}(,[0-9]{3})*)(\.[0-9]{1,2})?`），但要为每家银行/App 各写一套且易失效。**我们改为把通知原文丢给 AI（复用 7.4）解析**，泛化免维护，还能判断「是不是交易」过滤噪音。代价：AI 必需、文本发往用户自配端点（详见「统一管线」）。 |
| **隐私本地化是卖点** | 别人也强调「全程本地、不上传服务器」。这正是我们能强化的差异点。 |
| **连钱迹都长期拒做** | 行业标杆钱迹曾拒绝自动记账，认为「实时监控屏幕太激进、违背初衷」，改提供 Tasker 意图接口让用户自己触发 → 强化了「必须默认全关、纯 opt-in」的判断。 |

来源见文末。

## 方案总览

- **两条监听通道，都做，纯 opt-in**：NLS 打底（7.5a，先做，成本低、不用吓人的权限），无障碍补微信（7.5b，后做，需常驻保活）。
- **无障碍不做任何自动点击/操作，只读**；配置严格限定包名 + `canRetrieveWindowContent`。
- **截屏方案砍掉**——有这两条就不需要了。
- 两通道最终都产出同一个 `AiEntryDraft`，复用 7.4 的 `EntryDetailPage(initialDraft:)` 确认，**绝不自动落账**。

## 入口与信息架构

- 「我的 / 设置 → **自动记账**」独立页（`AutoCaptureSettingsPage`）。
- **默认全部关闭**，不影响普通用户（多数人不会用，我们自己也不一定用）。
- 页内让用户**选监听方式**：
  - 「开启通知监听（NLS）」——跳系统「通知使用权」授权；
  - 「开启无障碍自动记账」——跳系统无障碍授权 + 引导开常驻通知保活、电池白名单、自启动。
- 每种方式附：作用说明、覆盖哪些 App、隐私声明、**来源 App 白名单**（默认只勾常见几个）。
- **AI 为硬门槛**：解析依赖 AI（见「统一管线」），打开自动记账开关时若未配置 AI（`AiSettings` 无 baseUrl/apiKey），**引导先去 AI 设置页**配置（可填本地 Ollama/LM Studio 或云端）；未配好不允许开启。

## 通道分工

| 通道 | 覆盖 | 权限 | 代价 |
|------|------|------|------|
| **NLS 通知监听（7.5a）** | 支付宝、银行 App、云闪付（通知带金额） | 通知使用权 | 低；微信抓不全 |
| **无障碍（7.5b）** | 微信「支付成功」页 + 补充其他 | 无障碍（最重权限）+ 常驻通知保活 | 高；国产 ROM 保活难、需长期追版面 |

## 统一管线

```
[原生] NLS 收到通知 / 无障碍捕获支付成功页
  → 抽取「原始文本 + 包名 + 时间戳」封成 CapturedPayment
  → 写入原生 SharedPreferences JSON 队列（Flutter 引擎可能没在运行）
  → （可选）原生发一条「N 笔待记账」轻通知，点击拉起 App
[Flutter] 开屏 / 回前台 / 点通知 drainPendingCaptures（仿 consumeQuickEntryIntent）
  → 把攒着的多条原始文本批量喂给 AI（复用 7.4 的 requestAiEntryDraft，扩一个通知专用提示词）
  → AI 判断 isTransaction（过滤营销/聊天/系统噪音）+ 抽 金额/收支方向/对方/日期 + 归类到当前账本清单
  → AiEntryDraft → 存 pending_captures 表（SQLite，schemaVersion +1）+ 角标
  → 用户逐条 EntryDetailPage(initialDraft:) 确认/改 → 落账 → 删该 pending
```

- **解析用 AI，不写按银行/按 App 的正则**——这是相较正则最关键的取舍：正则要为每家银行、每个 App 各写一套且易失效，AI 直接读原文泛化，还能判断「这条到底是不是交易」过滤噪音（正则做不到）。原生只负责「捕获 + 入队」（薄），解析全在 Dart 侧复用 7.4。
- **AI 为必需**：自动记账 = 需先配 AI（见「入口」）。**主推本地 Ollama/LM Studio**——免费、且文本不出设备，让「AI 解析」也完全本地。
- **批量解析省调用**：开屏 drain 时把多条通知一次性喂给 AI，而非每来一条调一次，降成本与延迟。
- **AI 调用失败兜底**：把通知原文存成「待解析」草稿（只预填备注），用户手动补金额——数据不丢，零正则维护负担。
- **去重**：通知按 `key`+`postTime`；无障碍按「包名+金额+时间窗口」防同一笔重复捕获；落账前可一键忽略，规避「自动 + 手动」重复。
- `pending_captures` 为「未成账目的临时数据」，删账本/初始化清理，**不进 JSON 备份**。原生队列 drain 后清空。

## 国产 ROM 保活（无障碍通道的核心难点）

- 无障碍服务配 **前台服务 + 常驻通知**（即用户说的「常驻通知栏」），提升存活。
- 页内**引导用户**：加入电池优化白名单、开自启动（各家 ROM 路径不同，给文案指引）。
- **诚实告知**：国产 ROM 上仍可能被系统杀，自动记账**稳定性有限**，漏记时可用 7.2 导入 / 手动补。不承诺 100% 捕获。

## 隐私与合规红线

- **绝不自动落账**：一律进待确认队列，用户确认才成账。
- **默认全关、纯 opt-in、逐 App 白名单**。
- **文本发往用户自配 AI 端点**：自动记账靠 AI 解析，开启后支付通知/界面文本会**持续、被动地**发往用户配的 AI 端点（与 7.4「主动打字才发」不同，须在开关处明写）。**主推本地模型**（Ollama/LM Studio）——此时文本不出设备，AI 解析也完全本地，契合数据自主；用云端 Key 则按量计费、笔数多会累积，需告知。绝不上传截图。
- **无障碍只读不操作**：只 `canRetrieveWindowContent` 读文本，绝不 `performAction` 点击/输入；配置 `packageNames` 严格限定，不监听白名单外的 App。
- **ToS 灰色**：读微信/支付宝界面比读通知更敏感，页内明示风险由用户自行开启。

## 落地拆分

- **7.5a NLS 通知监听**：`AutoCaptureSettingsPage`（含 AI 未配置引导）+ 通知使用权授权 + 包名白名单 + 原生 `NotificationListenerService` 捕获入队 + AI 批量解析（扩 7.4 通知提示词，含 `isTransaction` 过滤）+ `pending_captures` + 复用 `initialDraft` 确认。i18n(zh+en) + 测试（提示词构造、草稿映射、drain、失败兜底）。
- **7.5b 无障碍补微信**：`AccessibilityService`（`onAccessibilityEvent` 按包名读「支付成功」页节点文本）+ 前台服务常驻通知保活 + 各 ROM 保活引导 + 复用同一管线。
- **截屏磁贴**：不做（进 Backlog 注记：NLS + 无障碍已覆盖，无需截屏）。

## 定案汇总

| 决策点 | 选择 |
|--------|------|
| 通道 | NLS（7.5a）+ 无障碍（7.5b），双通道纯 opt-in，无截屏 |
| 入口 | 「我的/设置 → 自动记账」独立页，默认全关，页内选监听方式 |
| 分工 | NLS 抓支付宝/银行/云闪付；无障碍补微信支付成功页 |
| 解析 | **AI 必需**，不写按银行/App 正则：原生捕获原始文本 → 批量喂 AI（复用 7.4 + `isTransaction` 过滤）→ 草稿。开关硬门槛：未配 AI 先引导设置。主推本地模型（文本不出设备）；调用失败存「待解析」草稿兜底 |
| 落账 | 一律产 `AiEntryDraft` 交 `EntryDetailPage(initialDraft:)` 确认，**绝不自动落账**（行业共识 + 我们红线） |
| 保活 | 无障碍配前台服务 + 常驻通知 + 电池白名单/自启动引导；诚实告知稳定性有限 |
| 无障碍边界 | 只读窗口内容、`packageNames` 严格限定，绝不自动点击/操作 |

## 开工前置（用真实样例调 AI 提示词与测试）

解析改用 AI 后，真实样例不再用于写正则，而是用于**打磨通知专用提示词、构造真机测试用例、验证 AI 识别率**：

- **通知**：支付宝、主流银行 App、云闪付「付款成功 / 到账」的真实通知 title/text 及来源包名（如 `com.eg.android.AlipayGphone`、各银行）。样例已收集（持续补充）：
  - 招商银行：`您账户0966与07月06日11:43收款人民币0.01`（支付宝转入招行卡，银行视角记「收款」，实为账户间转账 → 靠确认落账纠正）。
- **无障碍**：微信「支付成功」页的真实节点文本 dump（含金额/商户/时间字段）及包名 `com.tencent.mm`。

样例覆盖收入/支出/退款/转账各类后，据此定稿提示词并写测试；实测识别率与误报率达标后进入真机验收。
